# Web Flasher

> The web flasher lets you install the 9M2PJU ESP32 DX Cluster Client
> firmware directly from your browser. No software to install, no drivers,
> no PlatformIO, no command line. Just plug in USB and click a button.

The web flasher is hosted on GitHub Pages at:

**https://9m2pju.github.io/9M2PJU-ESP32-DX-Cluster-Client/**

This document explains how to use it, how to troubleshoot common issues,
how it works under the hood, and how the CI pipeline builds and publishes
it automatically on every push to `main`.

## What the web flasher is

The web flasher is a single web page (`docs/index.html`) that uses the
[ESP Web Tools](https://esphome.github.io/esp-web-tools/) library to flash
firmware onto an ESP32 board over a USB serial connection, entirely from
the browser. When you open the page, you see a list of the 12 supported
boards. Each board has an **Install** button. Clicking it opens a browser
prompt to select the serial port, then flashes the correct firmware
binary for that board.

The page itself is static HTML, CSS, and a small amount of JavaScript.
The actual flashing logic (Web Serial communication, erasing flash,
writing partitions, verifying) is handled by ESP Web Tools, which is
loaded from a CDN.

For each board, the page references a **manifest JSON** file that tells
ESP Web Tools which binary files to write and at which flash offsets. The
manifests and firmware binaries are generated by the CI pipeline and
published alongside the page on GitHub Pages.

## Browser requirements

The web flasher relies on the **Web Serial API**, which is only available
in Chromium-based browsers.

| Browser | Supported | Notes |
| :--- | :--- | :--- |
| **Google Chrome** | Yes | Recommended. Version 89+ on desktop. |
| **Microsoft Edge** | Yes | Version 89+ on desktop. |
| **Opera** | Yes | Chromium-based, supports Web Serial. |
| **Brave** | Yes | Chromium-based. |
| **Firefox** | No | Does not implement Web Serial. |
| **Safari** | No | Does not implement Web Serial. |
| **Chrome / Edge on Android** | Partial | Web Serial support on Android is limited; a desktop browser is recommended. |
| **iOS (any browser)** | No | iOS does not expose Web Serial. |

> **Requirements summary:** Use a desktop Chromium-based browser
> (Chrome, Edge, Opera, or Brave). Connect the board directly to the
> computer with a USB data cable, not through a USB hub if you can avoid
> it.

The page must be served over **HTTPS** (GitHub Pages does this
automatically), because Web Serial is only available in secure contexts.

## Step-by-step instructions

1. **Open the web flasher.**
   Go to https://9m2pju.github.io/9M2PJU-ESP32-DX-Cluster-Client/ in
   Chrome or Edge.

2. **Connect your board.**
   Plug your ESP32 board into your computer with a USB **data** cable.
   Avoid USB hubs if possible; plug directly into the computer.

3. **Find your board in the list.**
   The page lists all 12 supported boards with their chip and display
   details. Find the one that matches your hardware.

4. **Click Connect.**
   Click the **Connect** button on your board's card. The browser shows a
   serial port picker dialog.

5. **Choose the serial port.**
   Select the port that corresponds to your board:
   - On **ESP32-S3** boards, it often appears as
     `USB JTAG/serial debug unit`.
   - On classic **ESP32** boards with a CP210x or CH340 bridge, it appears
     as `CP210x USB to UART Bridge` or `USB2.0-Serial` respectively.
   - On macOS, the port name includes `usbserial-` or `cu.SLAB_USBtoUART`.
   - On Linux, it is usually `/dev/ttyUSB0` or `/dev/ttyACM0`.
   - On Windows, it is a `COMx` port.

   Click **Connect**.

6. **Click Install.**
   ESP Web Tools shows an install dialog. Click **Install**. You will be
   asked to confirm that the existing firmware on the board will be
   erased. Confirm to proceed.

7. **Wait for flashing to complete.**
   A progress bar shows the erase and write steps. The whole process
   takes about 30 seconds. Do not unplug the board during flashing.

8. **Finish.**
   When flashing completes, you can click **Connect** again to verify the
   install, or simply close the page and unplug the board. The device is
   now ready for [first boot setup](01-getting-started.md#first-boot-setup-mode).

> **After flashing:** The device boots into setup mode automatically
> (first boot, no Wi-Fi configured). Hold **BOOT** + press **RESET** at
> any time to re-enter setup mode and reconfigure.

## Troubleshooting

### The serial port does not appear in the picker

**Symptom:** You click Connect, but no port appears, or the board's port
is missing from the list.

**Fixes:**

- **Use a USB data cable, not a charge-only cable.** Many cheap cables
  only carry power. If the board powers on (backlight lights up) but no
  port appears, the cable is almost certainly charge-only. Try a cable
  that you know transfers data (e.g. one that works for flashing another
  device).
- **Plug directly into the computer**, not through a USB hub or dongle.
- **Close other programs** that may have the serial port open (Arduino
  IDE, PlatformIO monitor, a previous browser tab, `screen`, `picocom`).
  Only one program can hold the port at a time.
- **Refresh the page** and try again. Sometimes the Web Serial
  connection state gets confused.
- **Check browser permissions.** Make sure the site has permission to
  use serial ports. In Chrome, go to
  `chrome://settings/content/serialDevices` and verify the site is not
  blocked.

### ESP32-S3: the port is not detected at all

**Symptom:** An ESP32-S3 board (T-Display-S3, T-QT, T-HMI, Waveshare S3
Round, T-Watch S3, T-Display-S3 AMOLED) is plugged in but no port
appears, even with a known-good data cable.

**The BOOT button trick:** ESP32-S3 boards do not always enter the
bootloader automatically when plugged in. To force it:

1. Unplug the board from USB.
2. Hold the **BOOT** button down.
3. Plug the board into USB (keep holding BOOT).
4. Press and release **RESET** (if your board has one), or just keep
   holding BOOT for a couple of seconds after plugging in.
5. Release **BOOT**.

The board should now appear in the serial port picker as
`USB JTAG/serial debug unit`. Click Connect and proceed with flashing.

> **Why this happens:** On ESP32-S3, the USB OTG controller can enumerate
> in several modes. Holding BOOT on power-up forces the ROM bootloader to
> expose the USB serial interface so the host can see it. Once the
> firmware is flashed, the board's own USB CDC code takes over and the
> port appears normally on subsequent boots.

### Flashing fails partway through

**Symptom:** The progress bar starts but then errors out partway.

**Fixes:**

- **Do not unplug the board** during flashing. If you did, start over.
- **Try a different USB port** on the computer, ideally a port directly
  on the motherboard (back of a desktop) rather than a front panel port.
- **Disable browser extensions** that might interfere with Web Serial.
- **Try a different USB cable.** Intermittent flashing failures are often
  a marginal cable.
- **Try a different browser.** If Chrome fails, try Edge, or vice versa.

### The board flashes successfully but the screen is blank

This is not a web flasher problem; it is a board-specific issue. See
[Supported Boards](02-supported-boards.md) for board-specific notes. In
particular:

- **T-Watch 2020** requires AXP202 PMU initialization for the display
  backlight. The firmware handles this automatically, but if you flashed
  an older build it may not. Re-flash with the latest build.
- **T-Watch 2020 V3** uses backlight pin GPIO 15 instead of 12. If you
  have a V3, you need a custom build with `-D TWATCH_BL_PIN=15`. The web
  flasher ships the V1 pinout by default.

### The page loads but the Install buttons do not appear

**Symptom:** The board list is empty or the buttons do not render.

**Fixes:**

- **Enable JavaScript** for the site. ESP Web Tools requires JavaScript.
- **Check your browser version.** Update to a recent Chrome or Edge.
- **Check your network.** The page loads ESP Web Tools from a CDN
  (`unpkg.com`). If your network blocks `unpkg.com`, the buttons will not
  render. Try a different network or a mobile hotspot.

## How it works under the hood

The web flasher is built from three pieces:

### 1. The page (`docs/index.html`)

A static HTML page that:

- Renders a card for each of the 12 supported boards, with the board
  name, chip, and display details.
- Loads the ESP Web Tools install button component from
  `https://unpkg.com/esp-web-tools@10/dist/web/install-button.js` as an
  ES module.
- For each board, creates an `<esp-web-install-button>` custom element
  with a `manifest` attribute pointing to that board's manifest JSON.

The board list is defined in a JavaScript array inside the page, so
adding a board to the flasher is a matter of adding one entry to the
array and shipping a matching manifest.

### 2. ESP Web Tools

[ESP Web Tools](https://esphome.github.io/esp-web-tools/) is an
open-source library by the ESPHome project. It:

- Uses the **Web Serial API** to open a serial connection to the board.
- Implements the **Espressif ROM bootloader protocol** (slip framing,
  commands like `ESP_FLASH_BEGIN`, `ESP_FLASH_DATA`, `ESP_FLASH_END`)
  entirely in JavaScript.
- Reads the manifest JSON, downloads each listed binary file over HTTPS,
  and writes it to the correct flash offset.
- Optionally erases the flash before writing and verifies the written
  data afterwards.
- Provides the install dialog UI (progress bar, logs, error messages).

Because everything runs in the browser, no firmware binary ever touches a
server you control beyond GitHub Pages. The user's browser downloads the
binaries from GitHub Pages and streams them to the board over USB.

### 3. The manifest JSON

Each board has a manifest file at
`manifests/<board-id>.json` on GitHub Pages. The manifest tells ESP Web
Tools the chip family and the list of binary parts to write. Example
structure:

```json
{
  "name": "9M2PJU DX Cluster Client — lilygo-tdisplay-s3",
  "version": "1.0.0",
  "new_install_prompt_erase": true,
  "builds": [
    {
      "chipFamily": "ESP32-S3",
      "parts": [
        { "path": "firmware/lilygo-tdisplay-s3/bootloader.bin", "offset": 0 },
        { "path": "firmware/lilygo-tdisplay-s3/partitions.bin", "offset": 57344 },
        { "path": "firmware/lilygo-tdisplay-s3/boot_app0.bin", "offset": 57344 },
        { "path": "firmware/lilygo-tdisplay-s3/firmware.bin", "offset": 65536 }
      ]
    }
  ]
}
```

> The `offset` values above are illustrative. The actual offsets are
> generated by the CI pipeline and follow the standard Espressif layout:
> bootloader at `0x0` (ESP32-S3) or `0x1000` (ESP32), partition table at
> `0x8000`, `boot_app0` at `0xe000`, and firmware at `0x10000`.

The `parts` array always contains four entries for a full install:

| Part | Offset | Purpose |
| :--- | :--- | :--- |
| `bootloader.bin` | `0x0` (S3) or `0x1000` (ESP32) | ROM bootloader second-stage |
| `partitions.bin` | `0x8000` | Partition table |
| `boot_app0.bin` | `0xe000` | OTA partition selector (factory app) |
| `firmware.bin` | `0x10000` | The application firmware |

The `new_install_prompt_erase` field tells ESP Web Tools to prompt the
user to confirm erasing the flash before writing, which is the safe
default for a fresh install.

### Firmware binary packaging

The CI pipeline copies the four binary files for each board into
`site/firmware/<board-id>/`. The manifest's `path` values are relative to
the GitHub Pages root, so ESP Web Tools fetches
`https://9m2pju.github.io/9M2PJU-ESP32-DX-Cluster-Client/firmware/<board-id>/firmware.bin`
and so on.

## The CI pipeline

The web flasher is rebuilt and republished automatically by the
`.github/workflows/build-flash.yml` GitHub Actions workflow. It runs on
every push to `main` and on manual dispatch.

### Pipeline steps

1. **Check out** the repository.
2. **Set up Python** and install PlatformIO and `esptool`.
3. **Create `include/config.h`** by copying `include/config.example.h`
   (so the build has compile-time defaults).
4. **Build all 12 environments** with a single `pio run` (no `-e` flag
   builds every environment defined in `platformio.ini`).
5. **Prepare the site directory** (`site/manifests/` and
   `site/firmware/`).
6. **Copy `docs/index.html`** to `site/index.html`.
7. **Package firmware and generate manifests** using an inline Python
   script (details below).
8. **Upload the site directory** as a GitHub Pages artifact.
9. **Deploy** the artifact to GitHub Pages.

### The packaging script

The Python script inside the workflow does the following for each of the
12 board environments:

1. Looks up the board's chip family (`ESP32` or `ESP32-S3`) and bootloader
   offset (`0x1000` for ESP32, `0x0` for ESP32-S3) from a hardcoded map.
2. Finds `boot_app0.bin` inside the PlatformIO Arduino framework package
   (`~/.platformio/packages/framework-arduinoespressif32/tools/partitions/boot_app0.bin`).
3. Copies four files from the build directory
   (`.pio/build/<env>/`) into `site/firmware/<env>/`:
   - `bootloader.bin`
   - `partitions.bin`
   - `boot_app0.bin` (from the framework package)
   - `firmware.bin`
4. Builds the `parts` array with the correct offsets:
   - bootloader at the board-specific offset
   - partitions at `0x8000`
   - `boot_app0` at `0xe000`
   - firmware at `0x10000`
5. Writes the manifest JSON to `site/manifests/<env>.json`.

The board-to-chip map in the script is the source of truth for which
chip family and bootloader offset each board uses. If you add a new
board, you must add an entry to this map in the workflow as well as to
`platformio.ini` and `docs/index.html`.

### Deployment

The workflow uses the official GitHub Pages actions:

- `actions/upload-pages-artifact@v3` to upload the `site/` directory as a
  Pages artifact.
- `actions/deploy-pages@v4` to deploy the artifact to GitHub Pages.

The deploy job runs in the `github-pages` environment and outputs the
final page URL. Concurrency is controlled so that only one deploy runs at
a time, and in-progress deploys are cancelled when a new push arrives.

After the workflow completes, the updated page, manifests, and firmware
binaries are live at
`https://9m2pju.github.io/9M2PJU-ESP32-DX-Cluster-Client/`.

### Permissions

The workflow declares the permissions it needs:

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

`pages: write` and `id-token: write` are required by the GitHub Pages
deployment actions. `contents: read` is needed for the checkout.

## Local development of the flasher page

If you want to modify the flasher page (`docs/index.html`) and test it
locally, you need to serve it over HTTP(S) because ES modules and Web
Serial require a secure context. The easiest way is to run a local web
server:

```sh
# From the repository root, serve the docs/ directory
python3 -m http.server 8000 --directory docs
```

Then open `http://localhost:8000/` in Chrome. `localhost` is treated as a
secure context by Chromium browsers, so Web Serial works.

> **Note:** A local server will not have the `manifests/` or `firmware/`
> directories unless you also run the packaging script or copy them from
> a deployed site. For pure HTML/CSS/JS tweaks to the page layout, you
> can test without the manifests; the install buttons will fail to fetch
> the manifest, but the page layout and board list will render correctly.

To test a full end-to-end flash locally, build the firmware with
PlatformIO, copy the four binary files into a local `firmware/<env>/`
directory, generate a manifest JSON by hand (or adapt the CI script),
and serve the whole `site/` directory.
